## Overview

Lifecycle makes maturity explicit. Consumers should know whether a component is exploratory, adoption-ready or in a deprecation path before they build against it.

## Policy

Limia uses four lifecycle states across package exports and documentation:

## Workflow

Promotion should be evidence-based, not time-based. Moving from `beta` to `stable` requires proof that the component works in documentation, package API and product adoption.

- Accessibility checks pass for the intended interaction model.
- Documentation is complete in English and Portuguese.
- The component is used successfully in at least one production flow.
- Package exports and lifecycle files are aligned.

<Callout type="info" title="Governance rule">
  Docs, package lifecycle and app adoption should move together. A stable badge is not only a design judgment; it is a release contract.
</Callout>
