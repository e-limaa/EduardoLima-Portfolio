## Overview

Feedback is the pattern for showing progress, success, warning and failure across product flows. Good feedback reduces ambiguity and makes system state visible without interrupting users more than necessary.

## When to use

Use this pattern whenever the interface changes state because of user input, background processing or system conditions. Match the surface to the urgency: inline for local feedback, toast for transient global confirmation, alert for important contextual messaging, and dialog for blocking decisions.

<Callout type="info" title="Escalate intentionally">
  Start with the least disruptive feedback surface that still communicates the problem or result clearly.
</Callout>

## Composition

Loading feedback should live close to the action that triggered it. Success and error messaging should explain both the result and the next step when recovery is needed.

## Code

Compose feedback from Limia primitives and keep business logic in the product layer. Use alerts for contextual messaging, loading affordances near the triggering control, and toasts for non-blocking app-level confirmation.

```tsx
import {
  Alert,
  AlertDescription,
  AlertTitle,
  Button,
} from "@limia/design-system";
import { LoaderCircle } from "lucide-react";

export function SavePanel({ saving }: { saving: boolean }) {
  return (
    <div className="space-y-4">
      <Button disabled={saving}>
        {saving ? <LoaderCircle className="mr-2 h-4 w-4 animate-spin" /> : null}
        {saving ? "Saving changes" : "Save changes"}
      </Button>

      <Alert>
        <AlertTitle>Need attention</AlertTitle>
        <AlertDescription>
          Review the highlighted fields before publishing.
        </AlertDescription>
      </Alert>
    </div>
  );
}
```

## Accessibility

- Announce status changes in a way that matches urgency. Critical issues should not rely on toast alone.
- Keep loading feedback near the initiating action so keyboard and screen-reader users understand what changed.
- Write actionable error messages that help users recover instead of generic failure text.

<DoDontContainer>
  <DoDont type="do" title="Explain what happens next">
    Pair errors and warnings with the next action, such as retrying, editing a field or checking a connection.
  </DoDont>
  <DoDont type="dont" title="Hide progress behind motion only">
    Spinners and shimmer effects are supportive, but the label should still say what is loading or saving.
  </DoDont>
</DoDontContainer>
