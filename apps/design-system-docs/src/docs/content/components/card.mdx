## Overview

Cards group related content, controls and feedback into a single surface. They are useful when users need to scan or act on one entity at a time without losing the surrounding page context.

## Usage

Compose cards from the public slots instead of creating ad hoc wrappers: `CardHeader`, `CardTitle`, `CardDescription`, `CardContent` and `CardFooter`. Not every card needs every slot, but the structure should stay predictable when actions are present.

<DoDontContainer>
  <DoDont type="do" title="Group one subject per card">
    Keep each card focused on one object, task or summary so scanning remains fast.
  </DoDont>
  <DoDont type="dont" title="Stack too many responsibilities">
    If a card starts behaving like a full page with multiple sections, tabs and dense actions, split it into a dedicated screen or layout.
  </DoDont>
</DoDontContainer>

## Style

Cards are quiet structural surfaces. Use spacing, hierarchy and slot order to create rhythm before adding stronger decoration. Footer actions should feel subordinate to the card content, not compete with the page primary action.

<Callout type="info" title="Surface hierarchy">
  On dense pages, prefer one dominant card style and vary the internal content. Too many custom card treatments quickly turn a dashboard into a patchwork.
</Callout>

## Accessibility

- Cards are containers, not landmarks by default. Add semantic roles only when the content structure truly needs them.
- Keep headings meaningful so screen-reader users can understand each card when navigating by headings.
- If a whole card becomes clickable, provide clear focus treatment and an explicit accessible name.

<DoDontContainer>
  <DoDont type="do" title="Preserve readable structure">
    Use slot order to keep the heading first, body second and actions last whenever possible.
  </DoDont>
  <DoDont type="dont" title="Hide actions without context">
    Avoid icon-only footer actions unless the meaning is obvious and labelled for assistive technology.
  </DoDont>
</DoDontContainer>
