## Visão geral

Contributing define como mudanças entram no Limia sem transformar o design system em um depósito de necessidades específicas do app. O processo existe para proteger qualidade de API, disciplina de tokens e paridade documental.

## Política

Toda contribuição deve começar por um problema claro, e não apenas por uma ideia de implementação. Antes de adicionar ou alterar um componente, confirme se a necessidade é:

- um primitive reutilizável do pacote
- uma composição específica de produto
- um pattern apenas de documentação
- uma lacuna de token ou governança, e não de componente

<Callout type="info" title="Source of truth primeiro">
  As mudanças precisam entrar no source of truth do pacote ou dos tokens. Cópias locais no app e overrides pontuais não substituem trabalho de API pública.
</Callout>

## Fluxo

1. Enquadre o problema e confirme que o design system é a camada certa para resolvê-lo.
2. Defina API, estados, semântica e lifecycle alvo antes de codar.
3. Implemente em `packages/limia-design-system` ou `packages/limia-tokens`, não apenas nos consumidores.
4. Documente a mudança em inglês e português e atualize o metadata de lifecycle quando necessário.
5. Valide build, paridade e adoção antes de considerar a mudança pronta.

<DoDontContainer>
  <DoDont type="do" title="Mantenha a API pequena">
    Prefira composição, slots e variantes semânticas em vez de várias props criadas para uma única tela.
  </DoDont>
  <DoDont type="dont" title="Levar lógica de produto para o pacote">
    Fetch de dados, analytics, regras de roteamento e copy específica do app devem ficar fora do pacote do design system.
  </DoDont>
</DoDontContainer>
