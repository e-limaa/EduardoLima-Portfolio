## Overview

Buttons trigger actions, confirm decisions and move users through flows. Use them to submit forms, open overlays, confirm state changes or offer a clear next step.

## Usage

Use one primary action per region whenever possible. When several actions compete, reduce emphasis with `secondary`, `outline` or `ghost`. Reserve `destructive` for irreversible or high-risk actions.

## Style

Button styling communicates emphasis, not only brand. Size should reflect the surrounding density and the amount of text. Use `link` when the action should read as inline navigation instead of a surface-level CTA. Use icon sizes only when the action is widely recognizable or already labeled nearby.

Icons inside `Button` inherit the same semantic foreground token as the label, so filled variants keep icon contrast aligned with the active button foreground. In Limia, that behavior is tuned for the font-based `Icon` component rather than SVG icons.

## Code

Use `Button` directly from `@limia/design-system`. Avoid local wrappers unless you are composing product-specific behavior.

```tsx
import { Button } from "@limia/design-system";

export function Example() {
  return (
    <div className="flex gap-4">
      <Button>Create project</Button>
      <Button variant="outline">Cancel</Button>
    </div>
  );
}
```

## Accessibility

- Buttons are keyboard accessible by default and should remain focus-visible.
- Icon-only buttons must have an `aria-label` or nearby visible label.
- Destructive labels should be explicit, such as "Delete project" instead of generic "Confirm".
