## Overview

Color in Limia is semantic, layered and theme-aware. The system separates primitive scales from semantic roles so product code can express intent like `primary`, `layer-2` or `success` without coupling behavior to a specific hue.

## Principles

Limia follows the same core direction used by mature systems like Carbon: color is documented by role, theme and layer before it is documented by palette. That makes it easier to scale the system across surfaces without local forks.

<DoDontContainer>
  <DoDont type="do" title="Start from semantic roles">
    Prefer `bg-primary`, `bg-layer-2`, `text-muted-foreground`, `bg-success-subtle` and `border-warning-border` before reaching for a primitive scale.
  </DoDont>
  <DoDont type="dont" title="Treat the palette as the API">
    Primitive scales are source material for the system. They should not become the default authoring model in apps or shared components.
  </DoDont>
</DoDontContainer>

<Callout type="info" title="Themes and layers">
  Light and dark are not just inversions. Layers define how surfaces stack inside each theme so cards, popovers and embedded regions keep a clear depth model.
</Callout>

## Implementation

Use semantic classes in components and layouts, and reserve inline token consumption for cases where utility classes are insufficient. Primitive scales should stay mostly inside token definition work, data visualization and carefully reviewed brand moments.

<DoDontContainer>
  <DoDont type="do" title="Build with roles and layers">
    Use `bg-layer-1`, `bg-layer-2` and `border-subtle` to create depth. Use `bg-success-subtle` or `bg-warning-subtle` for support messaging that should remain contextual instead of dominant.
  </DoDont>
  <DoDont type="dont" title="Hardcode local palettes">
    Avoid arbitrary hex, one-off HSL values and app-level color forks in shared surfaces. They break theme parity and make lifecycle governance harder.
  </DoDont>
</DoDontContainer>

```tsx
import { Badge, Card } from "@limia/design-system";

export function Example() {
  return (
    <Card className="border-border-subtle bg-layer-1">
      <div className="space-y-3 rounded-lg border border-success-border bg-success-subtle p-4 text-success-subtle-foreground">
        <Badge className="bg-success text-success-foreground">Healthy</Badge>
        <p className="text-sm">
          Semantic support colors preserve meaning across light and dark themes.
        </p>
      </div>
    </Card>
  );
}
```

## Accessibility

Color should support meaning, never carry it alone. Pair support colors with text, iconography or structure, and treat contrast as a release gate for all themes and states, not only the default action palette.

<Callout type="warning" title="Contrast and redundancy">
  Never rely on color alone to communicate success, warning, error or selection. Pair it with copy, iconography, placement or another explicit state cue so low-vision and colorblind users receive the same signal.
</Callout>
