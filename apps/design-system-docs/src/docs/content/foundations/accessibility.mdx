## Overview

Accessibility is a system requirement, not a feature layer. Limia documentation, components and adoption guidance should all preserve inclusive interaction as part of the product contract.

## Principles

Limia follows the POUR model as an operational baseline:

- **Perceivable**: information must be presented in ways users can detect.
- **Operable**: interaction must work with keyboard and assistive technology.
- **Understandable**: labels, flows and errors must be clear and recoverable.
- **Robust**: semantics should survive across browsers, devices and assistive tools.

## Implementation

### Keyboard and focus

All interactive elements must remain operable with keyboard alone. Do not suppress the system focus ring without replacing it with an equally visible alternative.

### Semantics and labels

Prefer native HTML elements and add ARIA only when semantic text is missing or insufficient.

```tsx
<button aria-label="Close menu">
  <XIcon />
</button>
```

### Color, motion and feedback

- Meet WCAG AA contrast for standard text and component states.
- Never rely on color alone to convey status.
- Respect `prefers-reduced-motion` for decorative animation.
