## Overview

Radius softens the system and helps separate component roles. Shared radius tokens make nested surfaces feel related instead of visually accidental.

## Principles

Smaller controls should usually use tighter radii than the surfaces that contain them. Radius is part of component hierarchy, not an isolated decoration choice.

## Implementation

The system is anchored on `--radius`, then mapped into Tailwind-compatible radius utilities. This keeps buttons, inputs, cards and overlays visually coherent when the base value changes.

<DoDontContainer>
  <DoDont type="do" title="Nest radii intentionally">

  </DoDont>
  <DoDont type="dont" title="Mix unrelated shapes">

## Accessibility

Radius is mostly visual, but shape consistency improves recognition and reduces ambiguity between input fields, pills and dense surface groupings.
