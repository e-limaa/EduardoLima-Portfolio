## Overview

Contributing defines how changes enter Limia without turning the design system into a dumping ground for app-specific needs. The process exists to protect API quality, token discipline and documentation parity.

## Policy

Every contribution should start from a clear problem statement, not from an implementation idea alone. Before adding or changing a component, verify whether the need is:

- a reusable package primitive
- a product-specific composition
- a documentation-only pattern
- a token or governance issue instead of a component gap

<Callout type="info" title="Source of truth first">
  Changes must land in the package or token source of truth. App-local copies and one-off overrides are not acceptable substitutes for public API work.
</Callout>

## Workflow

1. Frame the problem and confirm that the design system is the right layer to solve it.
2. Define the API, states, semantics and lifecycle target before coding.
3. Implement in `packages/limia-design-system` or `packages/limia-tokens`, not only in consumers.
4. Document the change in English and Portuguese and update lifecycle metadata when relevant.
5. Validate build, parity and adoption before considering the change ready.

<DoDontContainer>
  <DoDont type="do" title="Keep the API small">
    Prefer composition, slots and semantic variants over adding many props for one screen.
  </DoDont>
  <DoDont type="dont" title="Ship product logic in the package">
    Data fetching, analytics, routing rules and app-specific copy should stay outside the design system package.
  </DoDont>
</DoDontContainer>
