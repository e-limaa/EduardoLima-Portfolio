## Visão geral

Versioning define como o Limia comunica risco de mudança para quem consome o sistema. O objetivo não é só higiene de release, mas upgrades previsíveis entre apps que dependem do pacote e dos tokens.

## Política

O Limia segue Semantic Versioning e trata comportamento visual, contrato de API e semântica dos tokens como parte da superfície versionada.

- `MAJOR`: mudanças quebrantes de API, exports removidos ou shifts visuais que exigem migração
- `MINOR`: adições compatíveis com versões anteriores, como novos componentes, props opcionais ou variantes
- `PATCH`: correções compatíveis, ajustes de documentação e melhorias internas

<Callout type="warning" title="Mudanças quebrantes">
  Breaking changes pertencem apenas a releases major. Uma release minor não pode alterar silenciosamente expectativas de layout, props obrigatórias ou o significado de um token semântico.
</Callout>

## Fluxo

Ao planejar uma release, avalie a mudança do ponto de vista de quem consome:

1. Um import, prop ou significado de token existente deixa de funcionar
2. Um default visual muda de forma que obriga ajustes de layout ou QA
3. Existe caminho de migração ou janela de depreciação quando necessário

Se a resposta for sim, a release deve ser classificada de acordo e documentada nas notas de lifecycle ou migração.
