## Overview

Versioning defines how Limia communicates change risk to consumers. The goal is not only release hygiene, but predictable upgrades across apps that depend on the package and tokens.

## Policy

Limia follows Semantic Versioning and treats visual behavior, API contract and token semantics as part of the versioned surface.

- `MAJOR`: breaking API changes, removed exports or visual shifts that require consumer migration
- `MINOR`: backward-compatible additions such as new components, optional props or variants
- `PATCH`: backward-compatible fixes, documentation corrections and internal improvements

<Callout type="warning" title="Breaking changes">
  Breaking changes belong only in major releases. A minor release cannot silently change layout expectations, required props or semantic token meaning.
</Callout>

## Workflow

When planning a release, evaluate the change from the consumer point of view:

1. Does an existing import, prop or token meaning stop working
2. Does a visual default change in a way that forces layout or QA updates
3. Is there a migration path or deprecation window when needed

If the answer is yes, the release should be classified accordingly and documented in lifecycle or migration notes.
