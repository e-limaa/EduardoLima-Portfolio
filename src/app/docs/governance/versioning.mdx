We follow [Semantic Versioning 2.0.0](https://semver.org/) (SemVer) to manage releases. This ensures that consumers of the Design System can upgrade with confidence.

## Version Format

Given a version number `MAJOR.MINOR.PATCH` (e.g., `1.2.3`), increment the:

1. **MAJOR** version when you make incompatible API changes.
2. **MINOR** version when you add functionality in a backward compatible manner.
3. **PATCH** version when you make backward compatible bug fixes.

## Deprecation Policy

When a component is slated for removal, we follow a strict multi-phase process to give teams time to migrate.

1.  **Mark as Deprecated**: Add a `@deprecated` TSDoc tag and a console warning in development mode.
2.  **Grace Period**: Provide a migration guide and keep the component for at least one MAJOR version cycle.
3.  **Removal**: Remove the component in the next MAJOR release.

<Callout type="warning" title="Breaking Changes">
  Breaking changes are **only** allowed in Major versions. Never introduce a breaking change in a Minor or Patch release.
</Callout>
